# PRD：X2MD v3.1 保存管线性能与架构改进

| 字段 | 内容 |
| --- | --- |
| 状态 | 待评审，未纳入 v3.1.0 发布实现 |
| 依据 | 2026-07-10 对当前运行主路径的代码复盘 |
| 范围 | `app/core/save.ts`、`app/core/profile-capture.ts`、`app/main/http-server.ts`、扩展保存请求 |
| 目标版本 | v3.2 起分阶段交付 |

## 1. 结论

当前保存链路功能完整，但存在明确的可优化点：图片下载串行、同步文件 I/O 占用 HTTP 处理、文件名碰撞检查非原子、保存历史读改写无串行保护，以及扩展采集入口的职责过度集中。它们在单次纯文本保存时影响有限，但在多图、批量抓取或多保存路径时会放大为等待、吞吐下降和偶发覆盖风险。

v3.1.0 不修改保存语义或存储格式；本 PRD 作为后续实现的可验收方案。

## 2. 现状与证据

```text
Extension → POST /save → savePayload()
  → localizeImages() → buildMarkdown() → 写入每个 save_path → 写 save_history.json
```

| 发现 | 代码证据 | 影响 |
| --- | --- | --- |
| 图片逐张下载和写盘 | `app/core/save.ts:78-91` 的 `for` 循环内 `await fetch` | N 张图片的等待时间约为各请求耗时之和 |
| 同步文件操作位于请求路径 | `mkdirSync`、`existsSync`、`writeFileSync` | 大文件或多目录写入可阻塞同一进程的其他 API 请求 |
| 冲突文件名仅精确到秒 | `timestamp()` 与 `existsSync()` 后的 `writeFileSync()` | 并发同名保存可能得到相同目标路径；检查与写入之间有竞争窗口 |
| 历史记录为读-改-写 | `appendSaveHistory()` | 并发保存可能丢失其中一条历史记录 |
| 采集 UI/流程集中 | `extension/content.js` | 页面适配、翻译、保存按钮和批量采集互相耦合，回归范围大 |

## 3. 目标与非目标

### 目标

1. 10 张图片保存的媒体阶段 p95 比当前串行基线降低 50% 以上，且不无上限并发。
2. 同时发起 20 个同标题保存时，生成 20 个不同 Markdown 文件，历史记录保留 20 条（受既有 20 条上限约束）。
3. 保存请求期间 `/ping` p95 小于 150ms（本机 SSD 基准，媒体下载不含在内）。
4. 对用户保持 Markdown、目录、配置与 HTTP 响应字段兼容。
5. 将扩展的保存编排与站点 DOM 适配拆开，给核心保存流程建立独立单测。

### 非目标

- 不改为云端保存、数据库或后台常驻队列服务。
- 不改变默认保存目录、Front Matter、图片链接格式或成功响应格式。
- 不在未验证前引入第三方队列/ORM/任务框架。

## 4. 方案

### P0：先建立基线与可观测性（1 个迭代）

- 增加仅本地 debug 开关：记录 `prepare_ms`、`media_ms`、`write_ms`、图片数量、目标目录数量和错误类别；默认不记录 URL/正文。
- 为纯文本、3 图、10 图、2 个保存路径、20 个同名并发保存建立可重复基准脚本与 fixture。
- 验收：CI 单测覆盖文件名唯一性和历史记录并发；基准报告保存到 `docs/acceptance/`，不提交用户内容。

### P1：安全的并发媒体本地化（1 个迭代）

- 以小型本地 `mapWithConcurrency(items, 4, worker)` 替换串行循环；并发数固定为 4，后续仅在基准证明需要时配置化。
- 保持输入图片顺序：worker 按索引写入结果槽位；失败仍保留原 URL，并写入现有 `image_localization_errors`。
- 为每个下载增加超时与 `AbortSignal`；超时视作单张失败，不中断 Markdown 保存。
- 写入先落临时文件再原子 rename，避免不完整附件被 Obsidian 索引。
- 验收：4 张受控延迟图片的总时长接近最长一批而非总和；顺序、失败回退和 Obsidian 嵌入语法回归测试通过。

### P2：异步且原子的 Markdown 与历史写入（1–2 个迭代）

- 使用 `node:fs/promises` 替换 `/save` 路径中的同步目录创建、写入和历史读写；不改变对外接口。
- 新增单一 `reserveOutputPath()`：优先使用原文件名，冲突时生成 `_<HHMMSS>_<随机短码>.md`，以 `open(..., "wx")` 原子占位，消除 TOCTOU。
- 为同一 `appDir` 的历史写入设进程内 promise mutex；锁内读取、合并、裁剪 20 条后以临时文件+rename 提交。
- 仅对文件系统错误返回现有 `errors`，每个保存目录继续独立成功/失败，保留部分成功行为。
- 验收：20 个并发同标题请求无覆盖；模拟第二个目录无权限时第一个仍成功；断言历史 JSON 始终可解析。

### P3：职责边界收敛（2 个迭代）

- 把 `extension/content.js` 迁为四个不改变行为的模块：`site-adapters/`（DOM 提取）、`capture-ui/`（按钮与状态）、`capture-flow/`（请求编排）、`shared/`（纯函数）。
- 把保存管线拆为 `prepareSavePayload`、`localizeImages`、`writeMarkdownCopies`、`saveHistory`；HTTP handler 仅负责认证、解析与调用。
- 以现有 fixture 为契约，先移动再调整；每次只迁一个站点适配器。
- 验收：原有扩展和 app 测试均通过；每个纯函数可在 Node 测试中独立调用；不引入新的运行时依赖。

## 5. 发布与回滚

- 分 P1/P2 独立发布，均保留 `X2MD_SAVE_PIPELINE_LEGACY=1` 的短期回退开关（仅开发/诊断使用，下一小版本移除）。
- 新实现连续两个版本无数据损坏或吞吐回归后删除旧路径。
- 任何写入异常、历史 JSON 损坏或附件残留超过基线即停止推进并回退上一实现。

## 6. 风险与约束

| 风险 | 控制措施 |
| --- | --- |
| 图片站点限流 | 并发固定为 4、超时、失败回退原 URL |
| 异步改造改变保存顺序 | 仅图片下载并行；返回的 `saved` 按配置路径顺序组织 |
| 原子占位留下空文件 | 捕获错误后清理占位；测试模拟写入失败 |
| 模块拆分引入站点回归 | 先无行为迁移、每个适配器配 fixture 回归 |

## 7. 成功定义

P0–P2 完成、通过完整 `npm run check`、保存基准达到第 3 节指标，并完成一次包含多图和并发同名请求的 macOS smoke 验证后，才将该 PRD 标记为已交付。
